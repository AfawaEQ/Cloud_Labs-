### Цель работы

1. Написать “плохой” Dockerfile, в котором есть не менее трех “bad practices” по написанию докерфайлов
    
2. Написать “хороший” Dockerfile, в котором эти плохие практики исправлены
    
3. В Readme описать каждую из плохих практик в плохом докерфайле, почему она плохая и как в хорошем она была исправлена, как исправление повлияло на результат
    
4. В Readme описать 2 плохих практики по работе с контейнерами. ! Не по написанию докерфайлов, а о том, как даже используя хороший докерфайл можно накосячить именно в работе с контейнерами.
    

### Ход работы
1. **"Плохой" Dockerfile**
```
FROM ubuntu

RUN apt-get update
RUN apt-get install -y python3 python3-pip

COPY . /app

WORKDIR /app

RUN pip3 install -r requirements.txt

CMD ["python3", "app.py"]
```
2. **“Хороший” Dockerfile**
```
FROM ubuntu:20.04

RUN apt-get update && \
    apt-get install -y python3 python3-pip && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

RUN useradd -m -s /bin/bash appuser

COPY requirements.txt /app/requirements.txt
COPY app.py /app/app.py

WORKDIR /app

RUN pip3 install --no-cache-dir -r requirements.txt

USER appuser

CMD ["python3", "app.py"]
```
3. **Сравнение**
	1. В плохом ```FROM ubuntu```, всегда используется последняя версия, при выходе обновления, есть риск поломки. В хорошем ```FROM ubuntu:20.04```, конкретная версия.
	2. В плохом по умолчанию работает от root, есть риск, что злоумышленник получит root-доступ к контейнеру (это плохо мягко говоря). В хорошем ```USER appuser```, даже если приложение будет взломано, злоумышленник не получит root-права внутри контейнера (это лучше).
	3. В плохом не очищаются лишние данные (```RUN apt-get install -y python3 python3-pip```), что замедляет работу. В хорошем это исправляется с помощью ```apt-get clean``` и ```RUN pip3 install --no-cache-dir -r requirements.txt``` (***--no-cache-dir***).
4. **Плохие практики по работе с контейнерами**
	1. Если запускать контейнер вот так: ```docker run -d myapp```. То контейнер не имеет ограничений по CPU и памяти. Если   Из-за утечки памяти или бесконечного цикла приложение "съест" все ресурсы , из-за чего может "уронить" хост-машину или другие контейнеры. Стоит использовать флаги ```--memory```, для ограничения максимальный объём оперативной памяти, и ```--cpus```, для ограничения количества ядер процессора. Запуск должен выглядеть примерно так: ```docker run -d --memory="512m" --cpus="0.5" myapp```
	2. Если запускать контейнер вот так: ```docker run -e DB_PASSWORD=supersecret myapp```. То, как несложно догадаться, наши "суперсекретные" данные можно легко получить. Стоит использовать ```echo "supersecret" | docker secret create db_password -```, таким образом ***supersecret*** будет сохранён в зашифрованном виде. Контейнер можно будет запустить с помощью ```docker service create --secret db_password myapp```, тогда наш ***supersecret*** будет расшифрован и вмонтирован в контейнер.
